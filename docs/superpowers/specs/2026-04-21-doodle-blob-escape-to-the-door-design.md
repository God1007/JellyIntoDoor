# 《涂鸦团子：冲进门里！》设计文档

副标题：`Doodle Blob: Escape to the Door`

## 1. 项目概述

本项目是一个单页浏览器休闲小游戏。玩家控制一只软乎乎、会变形的涂鸦团子，通过拖拽蓄力和松手弹射的方式穿过简短障碍关卡，目标始终是进入出口门。

游戏应当具备以下体验特征：

- 5 秒内理解操作
- 20 到 60 秒完成单关
- 失败后可立刻重试
- 手感软、弹、抖、糊，但仍然可控
- 视觉像一本会动的蜡笔速写本
- 完成度足够高，像一个真正的 casual web game，而不是技术演示

## 2. 产品范围

本次交付只做一个高完成度主模式：`Level Adventure`。

明确包含：

- 标题页
- 主模式关卡选择
- 10 个手工短关卡
- 关内计时、发射次数、星星收集
- 铜 / 银 / 金奖牌系统
- 最佳记录与关卡解锁
- 至少 3 种团子配色
- 1 套明确的进阶机制组：颜色能力
- 桌面与移动端支持
- 触摸与鼠标输入支持

明确不包含：

- 无尽模式
- 每日挑战
- 完整剧情系统
- 复杂角色成长树
- 高精度硬核平台跳跃设计

## 3. 核心设计目标

### 3.1 软体手感优先

团子不能像常规平台角色一样移动。乐趣来自于拖拽、压缩、反弹、挤压、抖动和轻度失稳。

### 3.2 读图清晰

每关目标永远只有一个：进门。路线必须能快速看懂，失败原因应易于理解。

### 3.3 短回合高重试率

单次尝试应很短，结算与重试应迅速，鼓励玩家刷更快时间、更少发射次数和全收集。

### 3.4 高风格一致性

玩法、UI、粒子、按钮、门、障碍、背景、音效都应统一到“纸张 + 蜡笔涂鸦”语言下。

## 4. 核心玩法循环

玩家进入关卡后，先观察一屏左右的地形和出口门位置，然后按住团子进行拖拽蓄力。团子会向拖拽反方向被拉长和压缩，松手后沿发射方向弹出，利用弹跳、挤压、斜坡动量、机关和临时能力进入门内。若失败，则可立即点击重试或自动快速重开，再次尝试更好路线。

循环结构：

1. 读路线
2. 拖拽蓄力
3. 发射与碰撞
4. 处理机关与能力
5. 进入门或失败
6. 结算 / 重试 / 下一关

## 5. 操作方案

### 5.1 主操作

- 按住团子：进入拖拽瞄准状态
- 拖拽：改变发射角度和力度，并让团子实时变形
- 松手：释放弹射

### 5.2 设计要求

- 不依赖键盘复杂输入
- 鼠标和触摸体验一致
- 操作规则必须在 5 秒内明白
- 拖拽越长，发射力度越强
- 发射角度和力度可视化清晰

### 5.3 输入反馈

- 蜡笔箭头指示方向
- 力度环或蓄力条指示强度
- 团子脸部变成“憋劲”表情
- 拖拽时出现纸屑线、草图辅助线和轻微抖边

## 6. 团子运动与“软体”实现

### 6.1 实现策略

采用“轻量刚体运动 + 视觉软体变形 + 表面状态修正”的混合方案，而非昂贵的真实软体物理。

### 6.2 运动层

运动层负责：

- 位置与速度更新
- 重力
- 基础碰撞
- 反弹与阻尼
- 地面、墙面、斜坡响应

这一层必须稳定、可预测、便于调参，避免真实物理引擎带来的漂移和难以控制的问题。

### 6.3 变形层

变形层独立于真实碰撞判定，负责表现：

- 静止时的轻微下垂与呼吸
- 拖拽时的拉长和压缩
- 发射前的蓄力挤压
- 落地后的 wobble 余震
- 高速撞击时的 squash / stretch
- 挤过窄缝时的压扁与回弹
- 失败时的软塌
- 过关时被门吸入后的开心抖动

### 6.4 手感原则

- 偏宽容而不是偏真实
- 允许轻微自动吸附到可站立平台
- 窄缝允许短时压缩通过，减少像素级卡死
- 墙面和斜坡会保留动量，鼓励利用惯性
- 高速失败也应好笑，不应显得生硬或惩罚性过强

## 7. 关卡系统与机制

### 7.1 基础机制

所有关卡都围绕以下机制组合展开：

- 基础弹射
- 地形反弹
- 挤过窄缝
- 按钮开门
- 弹簧板
- 风扇上升气流
- 动态小平台或轻微摇摆地形

### 7.2 进阶机制：颜色能力

颜色能力是本项目的唯一主进阶机制组，保证系统深度足够，但不会压垮 casual 可读性。

包含三种临时能力：

- `蓝色黏黏`：短时间可吸附在指定墙面上，便于挂墙后再次发射
- `绿色沉沉`：团子变重，可压按钮、触发重压机关或压碎脆弱地板
- `红色弹弹`：显著提高反弹和弹射收益，适合跨越大间隙或连续弹射

每种能力必须：

- 一眼看懂
- 有明显视觉状态变化
- 有简单清晰的使用目标
- 不叠加成复杂技能树

## 8. Adventure 模式关卡规划

关卡数量固定为 10 关，每关目标只有一个：进门。

### 第一组：认识团子

`1-1 起步涂鸦`
- 教玩家最基本的拖拽发射
- 直线或轻微抛物线进门

`1-2 门前小台阶`
- 学会通过反弹或二次落点登上矮平台

`1-3 挤挤小通道`
- 第一次通过窄缝，建立“团子可以挤”的认知

`1-4 星星顺手拿`
- 引入 2 到 3 个可选星星，鼓励非主线路探索

### 第二组：环境互动

`2-1 按钮开门`
- 团子坐上按钮，出口门打开，再找角度进入

`2-2 弹簧蜡笔`
- 用 spring pad 到达高处或远处平台

`2-3 风扇纸页`
- 学会利用 upward fan 与横向发射组合

`2-4 摇摆铅笔桥`
- 出现轻微动态地形，强调“有点不稳但很好玩”

### 第三组：颜色能力组合

`3-1 蓝色黏黏`
- 获取蓝色能力，学会短暂黏墙后再起跳

`3-2 绿色沉沉`
- 获取绿色能力，压按钮或压碎脆弱地板

`3-3 红色弹弹`
- 获取红色能力，通过强弹跳完成长距离移动

`3-4 毕业关`
- 组合按钮、窄缝、风区和颜色能力
- 路线依然清楚，不做高精度折磨

## 9. 收集、评分与重玩动机

### 9.1 收集物

每关放置 1 到 3 个 `蜡笔星星` 作为主要收集物。

用途：

- 提升完成度
- 触发全收集成就感
- 解锁额外配色或皮肤

### 9.2 评分指标

每关结算记录：

- 通关时间
- 发射次数
- 已收集星星数

### 9.3 奖牌系统

每关结算给出：

- 铜牌
- 银牌
- 金牌

判定依据由三个要素构成：

- 时间阈值
- 发射次数阈值
- 星星收集完成度

推荐规则是：只要满足更高档位的大部分条件即可给出对应奖牌，避免奖牌判定过于苛刻。

### 9.4 重玩驱动力

玩家应自然产生以下目标：

- 更快进门
- 用更少次发射完成
- 全收集星星
- 拿更高奖牌
- 试更滑稽或更顺的路线

## 10. 视觉风格规范

### 10.1 总体视觉语言

整体风格是“纸张上的蜡笔涂鸦世界”。不只是线条歪一点，而是从背景、地形、机关、团子、UI 到粒子都统一在同一视觉语言下。

### 10.2 画面元素规则

- 使用暖白或米白纸张底色
- 轮廓线粗、略不稳定
- 内部填色采用半透明涂抹与斜向笔触
- 形状保持轻微不规则，避免过于几何标准
- 局部使用重复重描和抖动，模拟手绘感

### 10.3 团子形象

团子必须具备：

- 简单可爱的圆形基础轮廓
- 两只清晰的眼睛
- 可变化的小表情
- 脸颊或轻微腮红
- 在拖拽、撞击、失败、过关时有明显表情变化

### 10.4 门的表现

门必须是关卡中最醒目的目标物之一。建议表现为快乐的小门或发光的贴纸门。

门的反馈包括：

- 可识别的待机发光或呼吸动画
- 开门状态变化
- 通关时吸入团子
- 星星或纸屑粒子庆祝

## 11. 动画与反馈

### 11.1 团子动画

- 待机呼吸 wobble
- 拖拽时拉伸与蓄力压缩
- 发射后的回弹摆动
- 落地后的 1 到 2 次余震
- 失败时软塌
- 通关时开心颤动

### 11.2 特效

使用统一的 doodle 粒子语言：

- 蜡笔碎屑
- 纸屑粉尘
- 冲击线
- 弹跳环
- 星星亮片
- 风流线

### 11.3 文字反馈

关卡内外可随机出现简短手写风反馈，例如：

- “软得刚刚好”
- “门很满意”
- “差一点点”
- “漂亮一弹”
- “糊过去了！”

文案语言以中文为主，整体语气轻松俏皮。

## 12. 音频方向

音频风格应轻巧、温和、不刺耳，不能破坏短时重试体验。

包含：

- 发射 `boing / 噗叽`
- 落地 `duf / 啪嗒`
- 收集短促闪亮音
- 通关上扬小音阶
- UI 翻页或纸张轻点音

背景音乐：

- 轻松、循环、不过分抓耳
- 适合连续重试
- 必须支持静音

实现方式允许使用轻量现成素材，或用轻量程序化 / 合成方式生成基础音效。

## 13. UI 与信息架构

### 13.1 标题页

显示：

- 中文标题《涂鸦团子：冲进门里！》
- 英文副标题小字
- 开始游戏
- 选关
- 颜色 / 皮肤
- 音效开关

### 13.2 选关页

风格类似涂鸦地图或手账页，显示：

- 10 个关卡节点
- 已解锁状态
- 奖牌结果
- 星星收集

### 13.3 关卡内 HUD

显示：

- 当前关卡编号或名称
- 当前星星数
- 计时
- 发射次数
- 暂停
- 重试

### 13.4 结算页

显示：

- 通关时间
- 发射次数
- 星星收集情况
- 奖牌
- 最佳记录
- 下一关按钮
- 重试按钮

### 13.5 UI 原则

- 信息层级清楚，不堆满屏幕
- 控件有手绘边框和贴纸感
- 出现和收起应有软弹或翻页过渡
- 中文文案统一，避免语言混杂

## 14. 技术方案

### 14.1 总体栈

- `Vite`
- 原生 `JavaScript`
- `HTML5 Canvas`
- DOM 叠加式 UI
- `localStorage` 存档

### 14.2 架构原则

- 运行简单
- 文件职责清楚
- 初中级开发者可读
- 方便桌面和移动端适配
- 不依赖重型游戏引擎

### 14.3 模块建议

- `index.html`：页面容器与 UI 根节点
- `src/main.js`：应用启动、场景切换、全局初始化
- `src/game/game.js`：游戏主循环与状态管理
- `src/game/blob.js`：团子运动、变形、表情、状态
- `src/game/input.js`：鼠标 / 触摸输入处理
- `src/game/physics.js`：基础碰撞与表面响应
- `src/game/level-data.js`：关卡配置数据
- `src/game/level-runtime.js`：关卡对象实例化、门、按钮、能力道具等逻辑
- `src/game/render/*`：纸张、地形、门、粒子、HUD 绘制
- `src/game/audio.js`：音效与静音控制
- `src/ui/*`：标题页、选关页、结算页、皮肤页
- `src/storage.js`：本地存档与记录

## 15. 存档与解锁

使用 `localStorage` 保存：

- 已解锁关卡
- 每关最佳时间
- 每关最少发射次数
- 每关星星收集记录
- 每关奖牌结果
- 当前选中配色或皮肤
- 音效状态

解锁策略：

- 默认解锁首关
- 完成当前关即可解锁下一关
- 额外配色或皮肤由累计星星或奖牌里程碑解锁

## 16. 响应式与平台支持

必须支持：

- 桌面浏览器
- 手机触摸屏

要求：

- 横屏体验优先
- 小屏幕上 UI 自动收紧
- 触控按钮尺寸足够大
- 不依赖 hover 才能完成核心流程
- 拖拽输入在移动端也必须顺畅

## 17. 测试策略

本项目不采用重型自动化关卡回放，而是采用“核心规则自动测试 + 手感人工调试”的组合。

自动测试覆盖：

- 发射角度和力度计算
- 奖牌判定
- 收集与结算统计
- 能力状态切换
- 机关开关状态
- 存档读写
- 关卡解锁流转

人工测试重点：

- 拖拽与松手在桌面 / 移动端的一致性
- 团子软体反馈是否有趣
- 失败后重试节奏是否够快
- 每关路线是否清楚
- UI 是否不会遮挡核心视野

## 18. 成功标准

项目完成后，应满足以下结果：

- 第一次进入游戏即可快速理解玩法
- 团子移动明显区别于普通平台角色
- 10 关都有明确主题且完成时间短
- 失败不会令人烦躁，反而鼓励重试
- 视觉风格统一、鲜明、可记忆
- UI、结算、选关、解锁形成完整小游戏闭环
- 作品足够完整，可作为创意编程或前端作品集项目展示

## 19. 实施边界总结

本项目优先追求：

- 手感
- 视觉表达
- 可重玩性
- 代码清晰度

本项目主动避免：

- 僵硬的普通平台跳跃手感
- 复杂键盘操作
- 过度真实的软体物理
- 过多模式导致的完成度下降
- 杂乱无章的 UI
- 看起来像原型而不是成品
